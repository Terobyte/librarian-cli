# Librarian `lib verify` — проверка и атрибуция цитат — Design

**Дата:** 2026-07-08. **База:** 0.1.1 (M7 отгружен: PyPI, MCP registry), спека v2.2,
M1–M7 закрыты. **Ревизия:** R1 после первого раунда критики (2026-07-10, три линзы:
контракты/UX, алгоритм/детерминизм, интеграция; все находки сверены с кодом).

## 1. Мотивация

LLM, цитирующий книгу, регулярно перевирает: склеивает фразы, «улучшает» формулировки,
приписывает не тому автору. У librarian уже есть всё, чтобы это ловить детерминированно:
побайтово стабильные главы + FTS5-индекс. `lib verify` закрывает нишу, которой нет ни у
одного book-MCP из конкурентного обзора M6 (2026-07-07: ebook-mcp, calibre-mcp,
book-mcp-server): **дословная проверка цитаты по первоисточнику**, без LLM и сети.
Перед анонсом перепроверить свежим поиском.

Два режима одной команды:

- **Проверка** (`--book` указан): «эта цитата действительно есть в этой книге?» —
  полный скан книги, без кандидатной фильтрации (recall ограничен только порогом
  вердикта и шагом окна, §3.2).
- **Атрибуция** (без `--book`): «откуда эта цитата?» — поиск по всей библиотеке через
  FTS5-кандидатов.

Позиционирование для README/анонса: *"An MCP server that catches Claude misquoting
your books."*

## 2. Принцип: ноль изменений в конвейере

`verify` — читатель готового выхода, как `find`/`get`. Конвейер ingest не знает о его
существовании.

- `PIPELINE_VERSION` не меняется; golden не регенерируются; проверка каждой задачи —
  `uv run pytest -q` зелёный без изменения golden. Файлы глав не переписываются:
  вся нормализация verify — в памяти, на лету, для обеих сторон сравнения.
- Читательские модули при этом трогаются: `search.py` получает новый read-only вход
  (кандидаты, §3.4), `catalog.py` — хелпер чтения главы (§6). Конвейер
  detect→…→emit — не трогается вообще.
- Версия пакета → 0.2.0 (новая фича, обратная совместимость полная).

```
library/ (готовые главы) ──→ verify.py ──→ вердикт + локация + diff
        ↑                        ↑
  режим книги: полный скан   режим полки: search.chapter_candidates (FTS5)
                             → топ-20 глав-кандидатов → тот же оконный проход
```

## 3. Ядро: `src/librarian/verify.py`

Чистые функции, без состояния, по образцу `catalog.py`. Публичный вход один:

```python
def verify_quote(lib_root: Path, quote: str, *,
                 book_id: str | None = None, limit: int = 3) -> dict
```

Ядро **всегда возвращает структуру** (§4.1) — включая случаи «пустая цитата» и
«короткая цитата в режиме полки» (`verdict: null` + `message`). Исключения из ядра —
только реальные ошибки: `LibError` (неизвестный book_id, sqlite без FTS5, занятый
индекс) и `ValueError` (`limit < 1`). Это снимает противоречие v0 дизайна, где CLI
требовал usage-исключение, а MCP — структурный ответ, из одного и того же входа.

### 3.1 Нормализация (цитата и текст глав — одинаково, в памяти)

Нормализация **токенизирующая**: канонический вид текста — последовательность слов,
склеенная одиночными пробелами. Пунктуация, кавычки, тире, многоточия, NBSP и
markdown-обвязка исчезают как класс, а не по перечислению. Порядок стадий:

1. **NFKC** (не NFC: `page.get_text` pymupdf по умолчанию сохраняет лигатуры
   `ﬁ ﬂ` — TEXT_PRESERVE_LIGATURES входит в TEXTFLAGS_TEXT, extractors/pdf.py:28, —
   а NFC их не разлагает; NFKC разлагает. Только внутри verify, вывод конвейера
   не меняется).
2. **Построчные фильтры** (формат глав — emit.render_chapter): удалить строки-заголовки
   `#…` (включая H1-титул главы) и строки-разделители `---`; снять лидирующие маркеры
   `> ` (блокквоты — эпиграфы и письма, самый цитируемый текст книги), `- ` (списки),
   `N. ` (нумерованные списки); маркеры снимать повторно (вложенные `> > `).
3. **Инлайн-сноски**: удалить маркеры вида `[N]` (emit рендерит `кит[1].`).
4. **casefold**, затем **ё→е** (после casefold Ё уже стала ё).
5. **Внутрисловная типографика**: `’ ‘ ′` → `'` (иначе `don't` ≠ `don’t`),
   `‑`(U+2011) `−`(U+2212) → `-`. Остальную типографику не конвертируем — она
   отпадёт на токенизации.
6. **Токенизация**: слово = максимальный прогон символов {буква, цифра, `'`, `-`}
   (категории Unicode, не ASCII); всё прочее — включая любые Unicode-пробелы
   (NBSP U+00A0, U+202F — переживают ingest: normalize.py схлопывает только
   `[ \t]+`), пунктуацию, кавычки «»""„, тире —–, многоточие … — разделитель.
   Токен без единой буквы/цифры (одинокое `-` от тире, `...`) отбрасывается.
7. Канон = `" ".join(tokens)`.

Следствия: «горят.» ≡ «горят»; `exact` не зависит от пунктуации, регистра, ё,
типографики — это и есть заявленное «типографика не в счёт». Ingest уже сделал за
нас NFC, CRLF→LF, soft hyphen и дегифенизацию — дублировать не нужно; NFKC поверх
NFC идемпотентно безопасен.

**Карта смещений.** На каждый токен при токенизации запоминается span оригинала
`(orig_start, orig_end)` — до всех удалений это тривиально, потому что стадии 2–5
реализуются одним проходом по исходным строкам с текущим абсолютным смещением
(строка либо выброшена целиком, либо у её хвоста известен сдвиг от снятого маркера).
Канон-строка ↔ токены ↔ оригинальные span'ы; `passage` и word-diff показывают
подлинный текст книги, а не нормализованный. Совпадение, примыкающее к удалённой
строке (заголовку), берёт границу по span'ам крайних **совпавших токенов** — заголовок
в passage не попадает.

### 3.2 Поиск совпадения

Единицы зафиксированы: **окна и границы — по токенам, скоринг — по символам
канон-строки**. Все `SequenceMatcher` — с `autojunk=False`: дефолтный autojunk на
последовательностях ≥200 элементов выбрасывает популярные элементы (для строк —
пробелы; проверено на проектном питоне: два 240-символьных текста дают ratio 0.0
при autojunk=True против 0.138 при False) — с ним пороги не значат ничего.
Цитата — вторая последовательность (`b`): её кэши `b2j`/`fullbcount` переживают
`set_seq1`, окно подставляется первой.

**Режим книги:** главы в порядке `n`. Для каждой главы — скользящее окно по токенам
шириной `L = len(quote_tokens)` с шагом `s = max(1, L // 4)` (последнее окно
прижимается к концу). Каскад на окно:

1. **Префильтр:** `quick_ratio()` канон-строк окна и цитаты ≥ **0.6**. Порог обоснован:
   при шаге `s` сэмплированное окно отстоит от истинного максимум на `s/2 ≤ L/8`
   токенов, т.е. теряет ≤ ~12.5% совпадения; худший интересный случай — истинный
   ratio на границе distorted 0.75 — даёт наблюдаемое ≥ 0.625 > 0.6. `quick_ratio` —
   документированная верхняя граница `ratio`, при autojunk=False префильтр строго
   sound. Допустимая оптимизация: инкрементальный пересчёт той же величины из
   посимвольных счётчиков при сдвиге окна — при инварианте «значение равно
   difflib.quick_ratio» побайтово ничего не меняет.
2. **Точный `ratio()`** для прошедших префильтр. Лучшее окно главы =
   argmax точного ratio, при равенстве — меньшее смещение.
3. **Уточнение границ** лучшего окна (если ratio ≥ 0.75): детерминированный
   hill-climb по токенам — в фиксированном цикле «расширить-влево → расширить-вправо
   → сузить-влево → сузить-вправо» шаг на один токен принимается только при
   **строгом** росте char-ratio; стоп — полный круг без единого принятия; границы
   не уходят дальше ±L токенов от сида. Строгое `>` исключает блуждание по плато.

**Одна глава — не более одной записи в `matches`:** лучшее окно (max ratio,
tie — min смещение). Цитата-рефрен (в golden `skazka` одно предложение повторено
~36 раз) даёт одну запись с минимальным смещением, а не 36.

**Режим полки:** значимые токены цитаты (§3.4) → `search.chapter_candidates` →
топ-20 глав `(book_id, n)` → тот же оконный проход только по ним. Метаданные
(title/author) кандидатов — из `read_book`, не из FTS-таблиц.

**Детерминизм ничьих:** сортировка `matches` — по `(−similarity, book_id, n)`;
similarity уже округлена (§3.3), так что предикат «равный similarity» в контракте
и в коде один и тот же. Внутриглавные ничьи разрешены на шаге «одна запись на
главу». Два одинаковых вызова обязаны давать побайтово идентичный `--json`.

**Производительность (порядок величин, не контракт):** книга 200k токенов, цитата
20 токенов → шаг 5 → ~40k окон; quick_ratio — O(длина окна) → секунды чистого
Python в худшем случае; точный O(L²)-ratio платят только окна выше префильтра.
Вырожденно-повторный текст (рефрен) прогоняет через ratio все окна главы — ограничено
размером главы, для v1 приемлемо; квоты и хитрые префильтры — parking lot (§7).

### 3.3 Вердикты

| verdict | критерий | смысл |
|---------|----------|-------|
| `exact` | равенство канон-строк | совпадение после нормализации (пунктуация/регистр/ё/типографика не в счёт) |
| `close` | similarity ≥ 0.95 | почти точная, расхождения показаны word-diff'ом |
| `distorted` | similarity ≥ 0.75 | это место, но цитата переврана |
| `not_found` | similarity < 0.75 | в книге/библиотеке такого нет |
| `null` | — | проверка не выполнялась: пустая (в т.ч. после нормализации) цитата; короткая цитата без `--book` (§3.4). Всегда с `message` |

`exact` — тест равенства последовательностей, не сравнение float (математически
`ratio()==1.0` ⟺ равенство, но равенство прямее и дешевле). `similarity` =
`round(char_ratio, 4)` — округление **в ядре**, до попадания в dict (прецедент —
quality.py округляет все скоры до 4 знаков); вердикт, сортировка и вывод используют
одно и то же округлённое значение — пользователь никогда не увидит «0.95» рядом с
`distorted`. Пороги 0.95/0.75 и префильтр 0.6 — стартовые константы в `verify.py`,
калибруются на фикстурах при реализации; в контракт и в тексты message числа
не зашиваются. Вердикт ответа = вердикт первого элемента `matches` (сортировка
§3.2); пустой `matches` → `not_found`.

Замечание для README: `close` при замене одного слова достижим на цитатах от ~150
символов; короткая цитата с заменённым словом честно уезжает в `distorted` — треть
трёхсловной цитаты и есть искажение.

### 3.4 Значимые слова и режим полки

Стоп-слов в кодовой базе нет — вводим замороженный мини-список константой в
`verify.py` (только для verify; поиск find не трогаем). Критерий: предлоги, союзы,
частицы, местоимения, связки. Стартовый состав (фиксируется юнитом):

- RU (~36): а, бы, в, вы, да, для, до, же, за, и, из, к, как, ли, мы, на, не, но,
  о, об, он, она, они, оно, от, по, при, с, со, так, то, ты, у, что, это, я
- EN (~33): a, an, and, are, as, at, be, but, by, for, from, he, i, in, is, it,
  no, not, of, on, or, she, so, that, the, they, this, to, was, we, were, with, you

Значимые токены = токены цитаты минус стоп-слова. Цитата **короче 5 значимых
токенов** без `--book` → `verdict: null` + message («слишком короткая цитата для
поиска по библиотеке — укажи книгу через --book»). Порог считается **после**
нормализации — цитата из одной пунктуации не даёт ложный `exact` на пустых
последовательностях.

FTS5-кандидаты — новая функция в `search.py` (реюз существующего `search()`
невозможен: он подмешивает до 3 книжных хитов с `n=None`, платит за сниппеты и
не отдаёт rank):

```python
def chapter_candidates(lib_root: Path, words: list[str], *, k: int = 20
                        ) -> list[tuple[str, int]]
```

Внутри: ленивый `sync()` (паттерн `search()`), затем chapter-only запрос —
`_build_match(words, "OR")` (реюз именно связки `_build_match`/`_escape`/`_stem_word`:
индекс хранит **сырой** текст, recall стемминга живёт на стороне запроса —
стем + префикс `"слов"*` + экранирование кавычек), `bm25(chapters, 0, 0, 5.0, 1.0)`,
и — обязательно — `ORDER BY rank, book_id, n LIMIT ?`: у SQLite нет стабильности
сортировки в контракте, граница топ-20 при равных rank должна быть пиновая.
Без сниппетов и highlight.

### 3.5 Word-diff расхождений

Для `close`/`distorted`: `SequenceMatcher(None, match_tokens, quote_tokens,
autojunk=False).get_opcodes()` по спискам токенов, каждая не-`equal` группа → пара
`{quoted: "...", source: "..."}` (пустая строка при вставке/удалении). Показ — по
оригинальным словам через карту смещений (span'ы токенов): пользователь видит
подлинную пунктуацию и регистр книги.

## 4. Контракты

### 4.1 Структура результата (единая для CLI `--json` и MCP)

```json
{
  "verdict": "close",
  "matches": [
    {
      "book_id": "bulgakov-master-i-margarita",
      "book_title": "Мастер и Маргарита",
      "author": "Михаил Булгаков",
      "n": 24,
      "chapter_title": "Извлечение мастера",
      "similarity": 0.9714,
      "passage": "— Рукописи не горят. — …",
      "diff": [
        {"quoted": "рукописи не горят никогда", "source": "Рукописи не горят"}
      ]
    }
  ],
  "message": null
}
```

- Номер главы — **`n`**, как везде в проекте (хиты find, list_chapters, book.json);
  `book_title`/`author`/`chapter_title` — те же имена, что в хитах find.
- `matches` отсортированы по `(−similarity, book_id, n)`; не больше `limit`
  (дефолт 3 — осознанно меньше find'овских 10: совпадение тяжёлое, passage+diff);
  `limit < 1` → `ValueError`; совпадения ниже порога not_found в список не попадают;
  не более одного совпадения на главу (§3.2).
- `verdict: null` → `matches: []`, `message` объясняет (пустая/короткая цитата).
- `not_found` → `matches: []`, `message` объясняет («ниже порога совпадения» — без
  числа, порог не в контракте; в режиме полки добавляет «FTS5-кандидаты могли не
  покрыть сильно искажённую цитату — попробуй --book»).
- `similarity` — округлённая в ядре до 4 знаков (§3.3), CLI и MCP видят одно значение.
- `passage` — подлинный текст книги (карта смещений), расширенный от span'а
  совпадения до границ предложений в оригинале; граница предложения = `[.!?…]` +
  пробельный символ (без обработки инициалов и сокращений — документированное
  приближение); жёсткий лимит **400 символов**: при упоре в лимит обрезка по границе
  слова + маркер `[…]` на обрезанном краю (маркер редактора, с текстом книги не
  спутать).
- **Байтовый контракт** — только у CLI `--json`: однострочный
  `json.dumps(res, ensure_ascii=False)`, как у `find --json` (cli.py:148; НЕ
  emit.canonical_json — тот для файлов библиотеки). MCP сериализует FastMCP —
  контракт там на уровне идентичного dict, не байтов.

### 4.2 CLI

```
lib verify "цитата" [--book ID] [--json] [--limit N]
```

Человеческий вывод (stdout, обвязка по-русски как весь CLI, enum-вердикт печатается
как есть — он часть контракта): вердикт, локация (книга/глава), similarity
(округлённая, 4 знака), пассаж, diff построчно:

```
distorted 0.8667 — bulgakov-master-i-margarita, глава 24 «Извлечение мастера»
  «…passage…»
  - цитировано: рукописи не горят никогда
  + в книге:    Рукописи не горят
```

Exit-коды (семантика grep, **осознанное отклонение** от общепроектного «1 = ошибка
выполнения» — зафиксировать в docs/MILESTONES.md как отклонение 38 при реализации):

| exit | случай | stdout |
|------|--------|--------|
| 0 | `exact` / `close` — цитата подтверждена | полный отчёт / JSON |
| 1 | `distorted` / `not_found` — не подтверждена | полный отчёт / JSON |
| 1 | рантайм-ошибка (`LibError`: нет FTS5, занят индекс, битая библиотека, **неизвестный book_id** — как у всех читателей: get/info/rm дают exit 1, вводить третье поведение нельзя) | **пустой** |
| 2 | usage: `verdict: null` (пустая/короткая цитата), `limit < 1` | пустой, message в stderr |

Дискриминатор для скриптов: exit 1 с непустым stdout = «проверено, не подтверждена»;
exit 1 с пустым stdout = ошибка (диагностика в stderr). В README прописать явно,
вместе с ловушкой: **exit 0 включает close** — строгий CI-гейт должен проверять
`verdict == "exact"` из `--json`. Данные → stdout, диагностика → stderr (инвариант
проекта).

### 4.3 MCP-тул `verify_quote` (шестой)

`serve.py`: тонкий адаптер `verify_quote(quote: str, book_id: str | None = None,
limit: int = 3) -> dict` — ноль своей логики, вызывает ядро. `verdict: null` /
`not_found` — **валидные ответы** с `message`, не исключения (паттерн `get
--budget`): MCP-клиент всегда получает структуру. Исключения тула — только LibError
(неизвестная книга, нет FTS5, занят индекс — действующий паттерн find/get_chapters)
и ValueError (limit).

Язык описаний: сейчас все пять описаний тулов — **по-русски** (serve.py:72–95;
v0 дизайна ошибочно считал их английскими). Аудитория MCP — международная (PyPI,
MCP registry), описания читает модель. Решение: в этой вехе **перевести все шесть
описаний на английский** (отдельный пункт плана §8; golden не задет). Новый тул —
с явным «use this whenever you quote a book to the user». `message`-поля ядра
остаются русскими — консистентно с get_chapters/find; полная локализация сообщений —
вне скоупа (§7).

## 5. Ошибки и края

- `book_id` — через существующий `validate_book_id` (traversal закрыт,
  MCP-безопасно). Неизвестный id → `UnknownBookError` → CLI exit 1 / исключение
  тула MCP.
- Пустая цитата и цитата, пустеющая после нормализации (одна пунктуация/markdown) →
  `verdict: null`, не ложный `exact` на пустых последовательностях.
- Цитата длиннее главы → окно = вся глава, сравнение целиком (`exact` недостижим
  автоматически: ratio < 1 при неравных длинах).
- Цитата через границу глав — не поддержана в v1; максимум — `distorted` по одной из
  половин. Документируем в README (Limitations).
- Пустая библиотека → `not_found` + message.
- `.search.db` нужен только режиму полки; **в CLI** режим книги работает и без FTS5.
  Оговорка: `lib serve` зовёт `sync_index` на старте (serve.py:102) и на sqlite без
  FTS5 не поднимется вовсе — «verify по книге доступен всегда» верно только для CLI;
  менять стартовое поведение serve в этой вехе не будем, формулировка в README —
  аккуратная.
- Книги в статусе `review` видны verify в обоих режимах (как и find) — фильтра по
  статусу нет; `failed` не сохраняются, края нет. `meta_locked` на verify не влияет.
- Блокквоты/эпиграфы, списки, сноски, NBSP, лигатуры PDF — закрыты нормализацией
  (§3.1), а не оговорками.

## 7. Тесты

- **Юниты** (`tests/unit/test_verify.py`): нормализация (кавычки/ё/тире/NBSP/
  лигатуры ﬁ→fi/апостроф don’t/markdown-маркеры `#` `>` `-` `N.` `[1]` `---`);
  карта смещений (расширения, удалённые строки, span'ы токенов); пороги вердиктов и
  округление similarity; tie-break (равный ratio → меньший offset; сортировка
  matches); «одна запись на главу» на рефрене; hill-climb границ (детерминизм цикла);
  word-diff; стоп-слова и порог «5 значимых»; `verdict: null` (пустая, пустая-после-
  нормализации, короткая без --book); `limit < 1`.
- **Интеграция на фикстурах:** синтетические `_mkbook`-книги (паттерн test_search;
  `_mkbook` уже продублирован в test_search/test_serve — вынести в conftest, третья
  копия не нужна) — golden-тексты вырожденные (рефрен, почти без типографики), для
  реалистичных кейсов не годятся, но `skazka` идеальна для теста рефрена и
  `> `-эпиграфа. Кейсы: точная цитата = `exact`; замена слова в длинном абзаце =
  `close` + diff ровно из одной пары; перестановка/пропуск клауз = `distorted`;
  чужой текст = `not_found`. Оба режима. **Shelf-режим — только на копии библиотеки
  в tmp_path**: ленивый sync создаёт `.search.db`, гадить в tests/golden/ нельзя
  (book-режим read-only — можно in-place).
- **Детерминизм:** два одинаковых вызова → побайтово идентичный `--json` вывод
  (байтовый контракт — CLI-путь, §4.1).
- **MCP-смоук:** `session.list_tools()` возвращает 6 (сейчас смоук list_tools не
  зовёт вообще — добавить); вызов `verify_quote` через stdio возвращает валидную
  структуру; тест — с `@pytest.mark.enable_socket` (asyncio socketpair, иначе
  падает под pytest-socket на Windows).
- **Golden:** не трогаются вообще (проверка — весь набор зелёный).

## 8. Вне скоупа v1

Мультиглавные цитаты; поиск перефразировок (семантика — принципиально не наша поляна:
нужен LLM); триграммный индекс для fuzzy-recall в режиме полки (доедем, если
FTS5-фильтр окажется узким местом); проверка по нескольким книгам с агрегацией;
квоты/инкрементальные префильтры производительности сверх §3.2; полная локализация
message-полей на EN; мягкий старт serve без FTS5.
