# Librarian M6 «поиск + MCP» — Design

**Дата:** 2026-07-07. **База:** pipeline 2.4 (commit f0682b2), спека v2.2, M1–M5 закрыты.

## 1. Мотивация

Инструмент существует «для работы с LLM», но последняя миля ручная: `lib get` → stdout →
копипаст в чат. M6 закрывает её двумя фичами (`lib serve` — из дорожной карты §20
спеки, «v3, за рамками»; `lib find` — новая, в спеке v2.2 её нет):

- **`lib find <query>`** — полнотекстовый поиск по главам всей библиотеки («в какой книге
  и главе про X?»).
- **`lib serve`** — MCP-сервер (stdio): Claude сам листает каталог, ищет и вытаскивает
  главы под токен-бюджет. Детерминированный RAG без эмбеддингов, без сети, без API-ключей.

Конкурентный контекст (проверено 2026-07-07): ebook-mcp, calibre-mcp (×5 вариаций),
book-mcp-server. Наши отличия: качество конвертации (дегифенация, колонтитулы, колонки,
скоринг), токен-бюджет отдачи, автономность (без Calibre/эмбеддингов/сети), FB2.

## 2. Принцип: ноль изменений в конвейере

Оба компонента — **читатели** готового выхода (`<id>/chapters/*.md`, `book.json`,
`index.json`). Пути глав берутся из `book.json` (`chapters[].file`, emit.py пишет
`chapters/NNN-….md`), а не глоббингом каталога.

- `PIPELINE_VERSION` остаётся `2.4`; golden не регенерируются.
- Проверка каждой задачи реализации: `uv run pytest -q` зелёный **без** изменения golden.
- ingest/reingest не знают о существовании поиска.

```
ingest (не меняется) ─→ library/ ─→ search.py ─→ .search.db (производный кэш)
                                        ↑              ↓
                          CLI find ─────┤        bm25 + сниппеты
                          MCP serve ────┘  (+ catalog/read_book/budget — уже есть)
```

## 3. Поиск: `search.py` + `<library>/.search.db`

**Движок:** SQLite FTS5 через stdlib-модуль `sqlite3`, офлайн. FTS5 — расширение
SQLite, в большинстве сборок CPython включено, но не гарантировано (системные Python
бывают без него); если нет — `LibError` по-русски («sqlite без поддержки FTS5 —
поиск недоступен»).

**Схема:**

```sql
CREATE VIRTUAL TABLE chapters USING fts5(
    book_id UNINDEXED, n UNINDEXED, title, text,
    tokenize = 'unicode61 remove_diacritics 2'
);
CREATE VIRTUAL TABLE books USING fts5(
    book_id UNINDEXED, title, author,
    tokenize = 'unicode61 remove_diacritics 2'
);
CREATE TABLE meta (key TEXT PRIMARY KEY, value TEXT);
-- meta: schema_version; per-book fingerprint (см. ниже)
```

**Почему две таблицы, а не колонки book_title/author в chapters:** название книги
заматчилось бы в каждой её главе — выдачу затапливает одна книга, лечить пришлось бы
оконными функциями/капами. Две таблицы: книжный хит («что у меня есть от Пелевина») —
одна строка на книгу, главный хит («где про маятники») — строка на главу. Обе таблицы
наполняются в той же транзакции синхронизации.

**Результат поиска — единая структура** (её же отдают `lib find --json` и MCP-тул
`find`): список хитов `{book_id, book_title, author, n, chapter_title, snippet}`;
у книжного хита `n` и `chapter_title` — null. Слияние: книжные хиты первыми (их мало
и они точнее отвечают на «какая книга»), затем главные по bm25; `limit` — на
объединённый список.

**Производный кэш вне контракта детерминизма.** `.search.db` — как `.lock`: исключается
из `tree_bytes` (правка conftest), пересобирается из markdown. Байты sqlite не обязаны
быть стабильными между версиями; результаты поиска детерминированы, потому что
детерминирован источник.

**Ленивая синхронизация (не хук в ingest).** Перед каждым запросом:

- fingerprint книги = sha256 байтов её `book.json` **+ `(size, mtime_ns)` каждого
  файла глав** (stat, без чтения контента). Хэш book.json ловит реингест (любая
  перезапись book.json меняет `ingested_at` = time.time(), emit.py) и ручную правку
  book.json (механика С-2 спеки v2.2: пользователь сам исправляет title/author в
  файле и ставит `"meta_locked": true` — CLI-команды для этого нет по дизайну);
  stat глав ловит прямую правку `*.md` мимо CLI — book.json хэша контента глав
  не хранит, одного его мало.
- **Слепая зона (документируем):** правка `.md` с сохранением size и mtime не
  детектируется. Спасательный люк — `--reindex`: полная пересборка индекса.
- Сверка множества книг + fingerprints с сохранёнными в `meta` → переиндексация только
  добавленных/изменённых, удаление ушедших. Свежий индекс = stat-проход, без чтения глав.
- `schema_version` не совпал → drop + полная пересборка.
- Конкурентность: **весь write-путь синхронизации — одна транзакция**
  (`BEGIN IMMEDIATE` … `COMMIT`), чтение — тоже в транзакции + `busy_timeout`;
  читатель никогда не видит частичный индекс. Гонка двух процессов в худшем случае —
  двойная пересборка, корректность не страдает. Сетевые ФС и так вне поддержки.

**Стоимость синхронизации.** Свежий индекс: stat-проход O(книги × главы) на каждый
запрос (CLI платит каждый вызов, serve — каждый `find`) — для персональной библиотеки
миллисекунды. Кэшировать fingerprints в памяти serve отклонено: stat-проход и есть
проверка свежести, кэш означал бы не замечать реингест во время MCP-сессии.
**Холодный старт:** первый `find` на непустой библиотеке строит индекс целиком —
на большой библиотеке первый MCP-вызов может упереться в таймаут клиента. Известное
ограничение v1; прогрев — любой первый `lib find <запрос>` в терминале; фоновая
пересборка — parking lot.

**Что индексируется:** в `chapters` одна строка = одна глава: `book_id`, `n`,
заголовок главы, полный markdown-текст главы; в `books` одна строка = одна книга:
название и автор из `book.json`. Книги в статусе `review` индексируются (они в
библиотеке).

**Запросы.** Пользовательский ввод — данные, не синтаксис: режем на слова по пробелам,
каждое слово → `'"' + w.replace('"', '""') + '"*'` — внутренние кавычки **удваиваются**
(иначе `o"brien` даёт несбалансированную строку и FTS5 падает), соединяем `AND`.
Сырой FTS5-синтаксис (NEAR, OR, скобки) не пропускаем. Пустой запрос после санитизации
→ usage-ошибка (exit 2). Нюанс unicode61: слово с дефисом (`well-known`) токенизируется
в два токена и в индексе, и в запросе — кавычки превращают его во фразу
`"well known"*`, поведение согласованное; документируем. Кавычка внутри слова работает
так же: `o"brien` → фраза соседних токенов `o`,`brien` — не только не роняет запрос,
но и находит (проверено на sqlite 3.50, включая префиксную форму `o"brie`).

**Ранжирование:** `bm25(chapters, 0, 0, 5.0, 1.0)` — заголовок весит больше текста;
`bm25(books, 0, 2.0, 1.0)` — название весит больше автора.
**Сниппеты:** главы — `snippet(chapters, 3, '«', '»', '…', 12)`; книжный хит —
название + автор с выделением совпадения (`highlight`).

**Ограничение (документируем в README):** стемминга для русского в FTS5 нет —
запрос «маятники» не найдёт «маятник»; префиксы смягчают обратное направление
(«маятник» найдёт «маятники»). Триграммный токенизатор отклонён в v1: индекс ×3–4,
ранжирование хуже.

## 4. CLI: `lib find <query> [--limit 10] [--book <id>] [--reindex] [--json]`

- Вывод (stdout, rich-таблица): `id · глава · заголовок · сниппет`. Ячейки книжного
  хита (совпадение в названии/авторе): глава `—`, заголовок = название книги,
  сниппет = название/автор с выделением совпадения.
- `--json` — хиты в stdout той же структурой, что у MCP-тула `find` (§3, один
  источник — search.py); пустой результат → `{"hits": []}`, exit 0, **без**
  stderr-сообщения (машинный вывод не смешиваем с человеческим).
- Ничего не найдено → сообщение в stderr, **exit 0** (пустой результат — не ошибка
  выполнения; §15 не меняется).
- `--book <id>` — фильтр по одной книге (WHERE book_id = ? поверх MATCH).
- `--reindex` — принудительная полная пересборка индекса (спасательный люк для
  слепой зоны fingerprint).
- Ошибки выполнения (нет FTS5, битая библиотека) → stderr, exit 1. Usage → exit 2.
- Библиотека пуста / нет каталога → как у `lib list` (та же диагностика).

## 5. MCP: `lib serve [--library <path>]`

**Стек:** официальный Python SDK (`mcp`), stdio-транспорт. Зависимость — **опциональный
extra** `librarian[serve]` (SDK тянет httpx/pydantic/anyio — ядро остаётся лёгким).
`lib serve` без установленного extra → русская подсказка «установите librarian[serve]»,
exit 1.

**Read-only в v1.** Никакого ingest через MCP: медленно (PDF — минуты), держит лок,
упирается в таймауты MCP-тулов. Parking lot.

**5 инструментов — тонкие адаптеры: не изобретают логику поверх CLI/search
(`find` — новая функциональность, но она целиком в search.py, MCP лишь адаптер):**

| Tool | Параметры | Отдаёт | Поверх чего |
|---|---|---|---|
| `list_books` | — | id, название, автор, статус, глав, токенов | index.json (как `lib list`, cli.py читает файл напрямую) |
| `list_chapters` | `book_id` | n, заголовок, токены, summary | read_book |
| `find` | `query, limit=10, book_id?` | хиты (книги и главы) со сниппетами | search.py |
| `get_chapters` | `book_id, spec? / budget?, from=1` | см. ниже | ядро `get` (catalog.py, §6) |
| `book_info` | `book_id` | та же проекция, что `lib info`: book + metrics/subscores/score/hard_triggers | общая с CLI функция проекции |

- **`get_chapters` — структурированный ответ, не плоский текст** (иначе Claude не
  узнает о продолжении на длинных книгах):
  `{text, chapters: [n…], next_from: n|null, message?}` — `next_from` = номер первой
  не вошедшей главы (null — книга отдана до конца).
- **Требование к интерфейсу ядра:** общая функция (§6, catalog.py) возвращает
  структурированный результат и в случае «первая глава не влезает» — пустой список
  глав с `next_from`, **не исключение**. CLI оборачивает пустой результат в
  ValueError/exit 1 (текущая семантика §15 сохраняется), MCP отдаёт как есть.
  Проверка пути главы (`is_relative_to`) живёт в ядре — MCP получает ту же защиту,
  что CLI.
- **`book_info` = проекция `lib info`, не весь report.json** — один источник правды
  о том, «что такое info»; `unknown_tags`/`removed`/`warnings` для LLM шум.
- **Дефолт budget-режима:** ни `spec`, ни `budget` не заданы → `budget=12000`
  (первый же «дай почитать» не должен падать). Оба заданы → ошибка тула, как в CLI.
- **Первая глава не влезает в бюджет** (CLI бросает ошибку): тул возвращает **не
  исключение**, а `{text: "", chapters: [], next_from: from, message: "глава N —
  X токенов, не влезает в бюджет Y"}` — Claude сам решит: поднять бюджет или взять
  главу через `spec`. Внутриглавная обрезка — вне v1 (parking lot).
- Корень библиотеки: `--library` / `LIB_HOME` / `./library` — как везде.
- `find` синхронизирует индекс перед запросом; остальные тулы читают файлы напрямую
  (всегда свежие).
- **Ограничение `book_info`:** значений `median_tokens` / `chapters_found` /
  `layout_flag` в report.json нет. `median_tokens` теряется сразу после расчёта
  субоценки structure (quality.py его никуда не пишет); `chapters_found` /
  `layout_flag` оставляют след только текстом в `hard_triggers` и только при
  срабатывании порога. Тул отдаёт report.json как есть, ничего не довычисляя.
- Описания тулов на русском (v1 для личного пользования; перевод — этап публикации).
- **Подключение (документируем в README):** extra по умолчанию не установлен —
  без него команда упадёт с exit 1. Самодостаточная строка:
  `claude mcp add librarian -- uv run --directory /path/to/librarian --extra serve lib serve --library /path/to/library`
  (`--extra serve` ставит mcp при первом запуске; альтернатива — заранее
  `uv sync --extra serve`). Claude Desktop — тот же stdio-запуск в JSON-конфиге.

## 7. Тестирование

- `test_search.py`: сборка индекса на fixtures-библиотеке; инкрементальная синхронизация
  (добавили/удалили/переингестили книгу, ручная правка meta_locked, **прямая правка
  `.md` — ловится по stat**); RU+EN запросы; префиксное поведение; кривые запросы
  (**кавычка внутри слова** `o"brien`, скобки, пусто); слово с дефисом; сниппеты;
  **книжный хит по названию/автору** (n=null, порядок слияния: книги перед главами);
  смена schema_version → пересборка; `--reindex`.
- `test_serve.py`: тулы in-process (прямой вызов хендлеров) — list/find/get roundtrip,
  spec×budget-валидация, **дефолтный budget без параметров**, **`next_from` на длинной
  книге**, **первая глава больше бюджета → структурированный ответ, не исключение**,
  несуществующий book_id. Хендлеры — обычные async-функции, зовутся напрямую через
  anyio pytest-плагин (anyio приезжает зависимостью mcp), stdio-транспорт в тестах
  не поднимается. Сетевой блок тестов не ослабляется (stdio/unix-socket разрешены).
  Плюс регресс на CLI: `lib get`/`lib info` после выноса ядра ведут себя побайтово
  как раньше (существующие тесты test_cli.py и есть эта проверка).
- `test_cli.py`: `find` — таблица, exit 0 при пустом результате, `--book`, `--reindex`,
  exit 2 на пустой запрос; `--json` — валидный JSON, структура совпадает с MCP-тулом,
  пустой результат `{"hits": []}` без stderr.
- Инвариант каждой задачи: golden побайтово не изменились (проверяется обеими точками:
  conftest.tree_bytes и test_golden, после дедупликации — одной).

## 8. Вне скоупа (parking lot)

- Публикация: отдельный публичный репо, EN README/описания тулов, лицензия, демо-GIF,
  листинг в awesome-mcp-servers. Отдельный заход после работающей фичи.
- ingest через MCP (прогресс, таймауты, локи).
- MCP-prompts (готовые слэш-команды «почитаем книгу» в Claude Desktop/Code) —
  украшение, тулов достаточно.
- Закладки / прогресс чтения — первое настоящее состояние вне «файлы + производный
  кэш», ломает философию; `next_from` покрывает продолжение в рамках сессии.
- Якоря до подзаголовка внутри главы (FTS5 offsets → маппинг на markdown).
- Фоновая пересборка индекса в serve (холодный старт первого find на большой
  библиотеке против MCP-таймаута клиента; в v1 — прогрев через терминал).
- Внутриглавная обрезка в budget-режиме (глава крупнее бюджета отдаётся частично).
- Русский стемминг / trigram-токенизатор.
- `--summarize` (LLM-аннотации, §20), OCR, MOBI.
- Числовые `median_tokens`/`chapters_found`/`layout_flag` в report.json (расширение
  схемы отчёта — отдельное решение, тянет bump PIPELINE_VERSION).
