# Librarian — история вех (M1–M7)

Сжатый журнал шести завершённых вех. Полные tero-планы (с встроенным исходником и
логами судей) удалены из рабочего дерева — они дублировали репо; их оригиналы живут в
истории git (`git log --oneline`, коммиты помечены ниже). Здесь остаётся то, что из
`git log` не восстановить: что каждая веха дала и **почему код расходится со спекой**.

- **Канон дизайна:** `librarian-spec-v2.2.md` (корень репо). `v2.1` устарела и удалена.
- **M6 (поиск + MCP):** `docs/superpowers/specs/2026-07-07-librarian-m6-mcp-search-design.md`
  оставлен как есть — компактный дизайн-документ, не раздут кодом.

## Сквозной инвариант всех вех

Детерминизм байт-в-байт: одинаковый вход + версия конвейера → побайтово идентичный
выход. Каждая веха обязана оставлять **golden зелёными без регенерации** — изменение
выходных байтов возможно только с осознанным `PIPELINE_VERSION`-bump.

| Версия | Веха | Почему поднята |
|--------|------|----------------|
| 2.2 | M1–M3 | базовая (совпадает с номером спеки) |
| 2.3 | M4 | полная схема `report.json` (§11.6), метрики качества |
| 2.4 | M6 | `.search.db` исключён из `tree_bytes` (производный кэш) |

Отклонения от спеки нумеруются **сквозно 1→39** через все вехи (живут в коде как
осознанные решения). Ниже — только консеквентные; полный список в истории планов.

---

## M1 «каркас» — pipeline, CLI, quality-заготовка
Коммиты `6e25f25`…`4263790` · отклонения 1–12 · PIPELINE_VERSION 2.2

Скелет всего конвейера: `detect → extract → normalize → structure → sections → quality →
emit`. Вендорёный токенизатор o200k_base (офлайн-контракт), TXT/MD-экстракторы,
canonical-JSON эмиссия с lock/publish/recover, `lib ingest/list/get/info/rm`, golden-тесты,
wheel-смоук установки.

Ключевые отклонения (живут в коде):
- **8 — частица «ка»:** `нау-ка → наука` (word-break), но побочно `ну-ка → нука`. Спека
  §6.1.2 даёт только suffix-list без исключения для «ка»; тест пинит оба поведения.
- **9 — детект кодировок:** charset-normalizer на коротком koi8-r даёт `shift_jis`; введён
  `_read_text` с кириллица+vowel-ratio эвристикой и перебором 6 кодировок.
- **2 — `make_id`** чист (без ФС); коллизионный суффикс `-{sha[:6]}` навешивает pipeline.

## M2 «FB2 + EPUB» — XML-форматы
Коммиты `4f980de`…`0dae414` · отклонения 13–16

Боевое использование `xmlsafe` (XXE off, no-network), zip-лимиты против bomb, экстракторы
FB2 (секции/эпиграфы/поэмы/сноски, zip-контейнер) и EPUB (spine-walk, DC-метаданные),
общий `html_blocks`, refine-фильтры R1 (мета-разделы) и R2 (печатное оглавление).

Ключевые отклонения:
- **16 — EPUB DocBook flat-nav:** DocBook-EPUB (O'Reilly/No Starch) метят главу и подразделы
  одним `<h1>`; под тройным гейтом (≥2 файла; ≥2 h1 в файле; ровно 1 TOC-запись) граница
  spine-файла = уровень 1, остальные заголовки `level += 1`. Плюс `_is_divider` для
  part-файлов (`## Part I` не проваливается внутрь главы). Самое сложное место кодовой базы.
- **14 — `xmlsafe.parse_html`:** голый UTF-8-кириллический фрагмент без decl давал мошибейк
  (lxml Latin-1-эвристика); добавлен `_decode_html` (BOM→decl→UTF-8) без ослабления XXE.

## M3 «DOCX + HTML» — офисные/веб-форматы
Коммиты `a63ea90`…`86cc721` + `bc37cb1` · отклонения 17–20

DOCX через mammoth→html_blocks (+core.xml метаданные), HTML через trafilatura с маппингом
в IR и счётчиком unknown_tags. `bc37cb1` — ревью-фиксы: **path-traversal в `get` через поле
`file` в book.json**, частичный merge конфигов, extract-timeout.

- **17 — `unknown_tags`** писался в report.json только непустым (чтобы не менять golden
  M1/M2); снято в M4 при переходе на полную схему §11.6.

## M4 «PDF + качество» — вёрстка и метрики
Коммиты `67e4f72`…`e6e3dc5`, `6f22c3e`, `021922f` · отклонения 25–33 · **PIPELINE_VERSION 2.3**

PDF-экстрактор (геометрия блоков, детект скан/зашифрован), проходы вёрстки P1–P7 (номера
страниц, колонтитулы, колонки, дефекты), полные метрики качества §11.1–11.6 (coverage/
structure/garbage/encoding/dehyphen → score → ok/review/failed), команда `lib doctor`.

Ключевые отклонения (все — риск-находки ревью, воспроизведены):
- **27 — P5.6** не сливает два подряд HEADING, если обе строки сами матчатся паттернам 6.1.3
  (иначе две главы схлопывались в «Chapter 1 Chapter 2» с потерей границы).
- **26/28 — P6** склеивает PARA только со следующим блоком чтения и пропускает сноски (иначе
  рвались цепочки 3+ страниц, сноска срасталась с телом следующей).
- **21 — PDF-фикстуры недетерминированы** (pymupdf пишет случайный `/ID`) — коммитятся
  байтами один раз; `make_fixtures.py` перезаписывает только с регенерацией golden.

## M5 «полировка» — эргономика и защита
Коммиты `9fd590c`, `65be5d5`, `f0ee2d3`, `ac6c75b`, `31c269a`, `1227d92`, `f0682b2` · откл. 22–24,29,34,35

`get --budget N` (жадные подряд главы под токен-бюджет), `reingest --all` (пересборка из
`source/`, стабильные id, `meta_locked`), spawn-guard таймаута экстракции (`f0ee2d3`),

- **34 — traversal guard в `get`:** budget-логика плана переписывала `get` наивным
  read-comprehension без проверки пути (уронило бы защиту BUG F-1); guard
  `ch_path.resolve().is_relative_to(...)` сохранён. Дыра позже закрыта в ядре (M6/T1).
- **22 — `LIB_EXTRACT_INPROCESS=1`** отключает spawn-guard в тестах (иначе +1с на ingest).

## M6 «поиск + MCP» — ридерские фичи
Коммиты `9ab88aa`, `aabc922`, `e098b75`, `f8f10a1`, `2efdc43` · отклонение 36 · **PIPELINE_VERSION 2.4**

Общее ридерское ядро в `catalog.py` (`validate_book_id` закрывает traversal для всех
читателей разом, `read_index`/`info_projection`/`get_chapters_core`), `lib find` — FTS5
полнотекстовый поиск по главам и названиям/авторам (bm25, сниппеты, стемминг RU/EN,
`--json`), `lib serve` — stdio MCP-сервер (5 read-only тулов поверх того же ядра, ленивый
импорт `mcp`, extra `librarian[serve]`).

- **36 — `parse_spec`** остаётся в `cli.py`; `get_chapters_core` берёт его локальным импортом
  внутри spec-ветки — разрыв цикла cli↔catalog (под `lib serve` точка входа — cli.py, модуль
  уже в `sys.modules`).
- `.search.db` — производный кэш (как `.lock`), вне контракта детерминизма, лениво
  синхронизируется по fingerprint книги (sha256 book.json + size/mtime глав).

## M7 «витрина» — публичный релиз v0.1.0, PyPI, MCP registry
Отклонение 37 · спека: `docs/superpowers/specs/2026-07-08-librarian-showcase-release-design.md`
(PIPELINE_VERSION не трогается — только раскладка/метаданные/доки/CI)

Репозиторий подготовлен как публичная витрина: `librarian/` сплющен из подкаталога в
корень репо (пакет и репо на GitHub переименованы в `librarian-cli`; импорт `librarian` и
команда `lib` не изменились); `pyproject.toml` получил PyPI-метаданные (readme, license,
authors, keywords, classifiers, `[project.urls]`); `README.md` (EN, канон для PyPI) +
`README.ru.md`; `LICENSE` (MIT), `CHANGELOG.md`; `examples/` с двумя книгами Project
Gutenberg и `docs/assets/demo.tape` (vhs-сценарий) для demo-GIF; `.github/workflows/
release.yml` — публикация в PyPI по тегу `v*` через Trusted Publishing (OIDC), без
секретов; `server.json` — заявка в официальный MCP registry (`io.github.terobyte/
librarian`, pypi-пакет `librarian-cli`).

- **37 — `mcp` переведён из опционального extra `[serve]` в основные зависимости, добавлен
  консольный скрипт `librarian-cli`** (форвардит argv в `serve`): схема MCP registry
  (`server.json`, 2025-12-11) не умеет выражать python-extras — клиент реестра запускает
  голый `uvx librarian-cli`, который обязан сразу поднимать сервер. Это отменяет решение
  M6 «лёгкое ядро», где `mcp` был extra-ом специально ради лёгкой CLI-only установки.
  Цена: CLI-only установка теперь тянет ~15 транзитивных зависимостей `mcp`
  (httpx/starlette/uvicorn/pydantic и др.). Extra `librarian-cli[serve]` оставлен пустым
  (`serve = []`) как совместимый шим — задокументированные однострочники с ним не падают.

## M8 «verify» — проверка и атрибуция цитат, релиз 0.2.0
Коммиты `230881e`…`375e5a4` (+ доки/версия) · отклонения 38–39 · спека:
`docs/superpowers/specs/2026-07-08-librarian-verify-quote-design.md`
(PIPELINE_VERSION не трогается — verify читает готовые главы, конвейер не меняется)

`lib verify "цитата" [--book ID] [--json] [--limit N]` — детерминированная проверка
дословности цитаты по библиотеке, без LLM и сети. Ядро `verify.py`: построчная
нормализация с картой смещений (заголовки/списки/блокквоты/сноски/NBSP/лигатуры — прочь,
спаны токенов — в оригинал), оконный матчер на `SequenceMatcher(autojunk=False)`
(квик-префильтр → точный ratio → hill-climb границ), вердикты exact/close/distorted/
not_found/null, word-diff по спискам токенов, passage с расширением до границ предложений
внутри сегмента (заголовок/`---` в passage не попадает). Режим книги — полный скан через
новый `catalog.chapter_text`; режим полки — FTS5-кандидаты через новый
`search.chapter_candidates` (OR-match, bm25). CLI-команда `verify` и шестой MCP-тул
`verify_quote` (выполняется в потоке — `anyio.to_thread`, не блокирует stdio-сервер);
заодно все 6 описаний MCP-тулов переведены на английский (были русские — v0 дизайна
ошибочно считал их уже английскими). Версия пакета → 0.2.0.

- **38 — exit-коды `verify` живут по семантике grep, не общепроектной «1 = ошибка
  выполнения»:** `exact`/`close` (цитата подтверждена) → 0; `distorted`/`not_found` (не
  подтверждена) → 1 с непустым stdout; рантайм-ошибка (`LibError`: неизвестный book_id,
  нет FTS5, занят индекс) → 1 с ПУСТЫМ stdout; usage (`verdict: null` — пустая/короткая
  цитата, `--limit < 1`) → 2. Дискриминатор для скриптов — пустота stdout при exit 1, а
  не сам код. Осознанное отклонение: у остальных команд `1` всегда означает ошибку
  выполнения; verify по смыслу ближе к `grep` (нашёл/не нашёл — тоже результат, не
  ошибка).
- **39 — режим полки verify расширяет FTS5-кандидатов е→ё-вариантами сверх спеки §3.4:**
  FTS5 `unicode61` не сворачивает ё→е для кириллицы (`remove_diacritics` работает на
  латинице/диакритике, не на кириллице — эмпирически проверено критиком плана: `MATCH
  'котел'` не находит `'котёл'`), а канон цитаты в verify уже ё-сложен (§3.1). Без
  расширения дословная цитата с «е» вместо «ё» промахивалась бы мимо своей книги в
  режиме полки. Фикс — забота `verify.py`, не `search.chapter_candidates`: для каждого
  значимого токена подставляются варианты с одиночной заменой е→ё в каждой позиции (кап
  8 вариантов на слово, детерминированный порядок: оригинал, затем подстановки слева
  направо); сама `chapter_candidates` остаётся дословно по спеке (`words → OR-match`).
